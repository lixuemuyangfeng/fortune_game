# Phaser 3 小游戏需求与规格说明

## 1. 项目定位

《暴富幻想所》应从当前 Vite + DOM 找茬原型，升级为 Phaser 3 驱动的交互型小游戏。目标不是做一个信息面板网页，而是做一个有命中反馈、角色状态、场景动效、关卡节奏和结算仪式的黑色幽默小游戏。

当前 DOM 原型可继续作为剧情、线索、热区和文案参考，但最终游戏画面应由 Phaser 3 的 Canvas/WebGL 场景承载。DOM 仅保留外层容器、调试入口、平台桥接或必要的分享/排行壳层。

## 2. 核心体验目标

- 玩家进入关卡后，首先感到自己在一个荒诞但统一的“处理室/现场”里，而不是在运营后台。
- 线索可以藏得不明显，但必须是明确物件，不能靠空白屏幕、模糊 UI 或抽象色块充当答案。
- 点击命中要有局部动效、音效或震动反馈，且反馈只在对应物件附近出现。
- 角色状态要通过正式素材变化表达，不能用 CSS 在 raster 人脸上画眼睛、嘴角、腮红或表情线。
- 每找到一个新线索，场景里的主角或代理角色必须出现可感知的进度状态变化，不能只更新左侧文字、进度条、灯光或机器。
- 通关后要收束在证据、机器、下一关入口和一句黑色幽默反馈上，不展示资源卡、处理线、排行榜等后台系统。

## 3. 技术路线

### 3.1 引擎与运行层

- 使用 Phaser 3 作为主游戏引擎。
- 保留 Vite + TypeScript 工程形态，新增 Phaser scene 层。
- 游戏主画面使用 Phaser Canvas/WebGL 渲染。
- UI 优先使用 Phaser 内部 UI/容器绘制；平台分享、微信/抖音桥接、调试面板可保留 DOM。
- 首阶段迁移第一关，后续再迁移朋友圈和财神庙。

### 3.3 状态管理

- 继续复用现有 `src/core/config.ts` 的关卡、证据、进度概念，但应逐步拆出游戏专用配置。
- 每个关卡至少包含：背景、线索 sprite、热区、命中动效、角色状态、机器状态、完成文案。
- 保存字段应覆盖：关卡进度、已找到线索、当前关卡、广告观看记录、排行榜模拟/平台数据。

## 4. 素材规格

### 4.1 素材类型

- 背景：`WebP` 或 `PNG`，建议 16:9，至少 1920x1080 母版。
- 线索：透明 `PNG/WebP` sprite，独立于背景，便于点击、描边、抖动和隐藏/显示。
- 角色：分层 sprite、sprite sheet、Spine/DragonBones 或多状态透明 PNG。
- 动效：sprite sheet、粒子、短序列帧、透明视频，优先用 Phaser 粒子和 sprite sheet。
- UI：可使用 Phaser 图形、九宫格面板、bitmap/font text；避免 DOM 卡片堆叠成后台感。

### 4.2 角色状态

所有关卡都必须把角色状态当作核心玩法反馈，而不是装饰。每关先确定“被拯救/被污染/被处理”的主角或代理角色，然后按线索进度准备状态资产。

通用规则：

- 若关卡有 `N` 个线索，必须准备至少 `N + 1` 个角色状态：`progress-0` 到 `progress-N`。
- `progress-0` 是开局状态，`progress-N` 是完成状态，中间每个状态对应找到一条新线索后的状态。
- 每次新增线索后必须能看到表情、眼神、肩膀、手部、站/坐姿或身体重心变化。只改变文案、进度条、机器灯、环境色温或音效不算角色状态变化。
- 角色必须是独立场景单元：透明人物 spritesheet、分层人物、Spine/DragonBones、透明视频，或和背景完全匹配的完整头肩/全身 overlay。
- 不得移动包含背景物体的矩形裁剪块。如果人物从背景里拆不出来，必须先生成 clean background、人物层和 foreground occluder，再做动画。
- 状态变化要克制但可读：idle 循环可以是呼吸、低头、肩膀沉；命中瞬间可以是抬眼、惊醒、松动；完成态可以是长出一口气或姿态回正。

第一关周启明的基准状态为 6 个进度状态，让每次新线索加入时都能看到表情或身体语言变化：

- `progress-0`: 0/5，疲惫、失魂、肩膀垮，默认底图或人物层。
- `progress-1`: 1/5，被第一条线索惊醒，眼睛或眉头有明显反应。
- `progress-2`: 2/5，开始回焦，手部或头部姿态能看出在思考。
- `progress-3`: 3/5，肩膀略微回正，眼神更清醒。
- `progress-4`: 4/5，出现谨慎松动或轻微释然。
- `progress-5`: 5/5，松一口气、表情自然放松、场景略回暖。

禁止只覆盖脸部五官。若使用 overlay，必须覆盖完整头肩或完整人物局部，并与原图光照、边缘、姿态一致。贴合不自然时宁可不用。

角色状态是第一关从 web 演示变成游戏的硬门槛。只接入 Phaser、点击热区、音效、粒子或环境光，不算角色反馈完成；如果周启明仍是背景图里的静态人物，本阶段只能标记为“Phaser 骨架”，不能标记为“可验收游戏切片”。

这个标准适用于后续所有关卡：如果该关主角/代理角色仍是静态背景，或者只用 UI 动效表达进度，该关只能标记为“交互骨架”，不能标记为“可验收游戏切片”。

### 4.3 第一关素材清单

- 办公室背景，人物和线索最好分层。
- 周启明 `progress-0` 到 `progress-5` 六个进度人物层或 spritesheet。
- 工位还魂机：idle、pulse、stable 三状态。
- 5 个线索 sprite：亏损曲线、金价手机、owner 消息、花呗便利贴、刮刮泪。
- 5 类命中特效：K 线抖动、金线扫光、聊天戳章、纸片弹起、刮痕扫光。
- 完成态证据袋/封存状态。

## 5. UI 与视觉规范

### 5.1 总体气质

界面要像一个荒诞处理室，不像 SaaS 后台、数据看板或营销落地页。视觉基调为深绿处理室、暖黄任务反馈、低饱和办公室光、纸张和票据材质。避免紫蓝科技渐变、扁平 SVG 拼贴和大面积运营卡片。

### 5.2 第一屏信息

第一关第一屏只展示：

- 品牌：`暴富幻想所`。
- 地点与时间：`字节跳桶开放办公区`、`周三 15:27`。
- 关卡名：`键盘声变轻了`。
- 当前目标：找齐 5 个让工位失魂的诱因。
- 右侧/中心游戏场景。
- 开始按钮与可选提示按钮。

不要提前展示后续关卡档案、资源、榜单、处理线或设施升级。

### 5.3 任务与进度

- 任务区域要一眼可读：目标标题、`0/5`、短进度灯位即可。
- 不使用英文 `MISSION/CLEAR`。
- 不使用笨重方格槽、后台步骤条或运营指标卡。
- 找到线索后，用证据板、便签、票据、钉子等游戏化容器收纳，不要排成普通标签列表。

### 5.4 完成态

完成态只保留：

- 场景完成反馈。
- 已找到证据板。
- 证据袋已封口。
- 下一关入口。
- 一句黑色幽默结算文案。

处理线、好友榜、资源产出、设施升级、今日人格等系统不得直接出现在第一关完成页。它们可以作为后续主菜单、店铺页或关卡外系统，但不能抢走关卡结算节奏。

### 5.5 视觉铁律

以下规则优先级高于实现便利性。没有素材或没有设计把握时，宁可少做，也不能用临时图形污染场景。

- 禁止假叙事：画面没有真实展示的动作，不得写进文案。例如没有喝水动画、杯子状态或人物动作时，不得说“先喝水”。
- 禁止占位几何体：不得把半透明圆、矩形、白条、黄条、色块覆盖在 raster 场景上充当角色、杯子、光效或反馈。只有它被明确设计成纸张、票据、屏幕、印章、机器灯、扫光等场内物件时才可使用。
- 禁止补丁式人物反馈：没有正式人物状态素材时，不得在人物身上覆盖幽灵身体、脸部五官、透明头像或表情符号。可接受的降级方案是环境色温、机器状态、局部灯光或证据板变化。
- 禁止背景块动画：不得把包含桌子、同事、窗户、屏幕或其他远景元素的矩形人物区域拿来抖动、缩放、位移。角色动效必须作用在透明人物层或骨骼/视频人物层上。
- 禁止不明所以的动效：每个命中特效必须回答“这个物件为什么这样反馈”。如果解释需要写在代码注释或交付说明里，说明视觉失败，应重做。
- 禁止反馈大于物件：除非目标物本身很大，否则反馈不得比目标物更抢眼，不得遮挡人物、机器、其他线索或主要阅读区域。
- 禁止文字脱靶：反馈文字必须和线索语义一致。老板/owner 聊天的反馈应表达“催办被截获、压力被标记、需求排队”等工作压迫语义，不得出现与聊天无关的“已归还”等泛化词。
- 禁止为了丝滑牺牲含义。动效的首要目标是命中确认和叙事表达，其次才是流畅度。
- 禁止把引擎迁移当作游戏化。主角没有真实表情或动作变化时，不得把 Phaser 版本描述为“更像游戏”；只能说完成了运行层和交互骨架。
- 每次改动游戏场景或 UI 后，必须人工检查 Playwright 截图。出现透明水杯、幽灵白条、大黄条、莫名标签、遮挡主体、文字裁切或 hover 暴露答案时，视为验收失败。

## 6. 交互与动效规格

### 6.1 热区规则

- 默认状态不显示热区框、红圈、遮罩或高亮。
- 鼠标 hover 不暴露答案。
- 点击命中后，在命中物件附近播放短反馈。
- 主动提示可以短暂显示目标范围，但必须由玩家触发。
- 已找到状态只保留小而明确的标记，不遮挡其他线索。

### 6.2 命中反馈

第一关命中反馈：

- 亏损曲线：屏幕内红色折线短促抖动。
- 金价手机：只在手机屏幕范围内出现细窄金线或数字闪烁，不得出现贯穿手机的大黄条。
- owner 消息：目标消息轻震，并出现小型“催办截获”或“需求排队”类戳章；不得使用与聊天语义无关的“已归还”。
- 花呗便利贴：便利贴自身轻微弹起或边缘抖动，不得叠加透明黄色大框。
- 刮刮泪：票面刮痕局部扫光，光效必须贴在彩票票面内。
- 工位还魂机：每找到一个线索短促亮一下，5/5 后波形稳定。

### 6.3 角色反馈

- 角色状态变化必须来自人物素材状态切换、sprite sheet、骨骼动画或完整 overlay。
- 不能用 CSS/Canvas 简单线条直接画在 raster 脸上。
- 每个新线索进入证据板时，角色必须同步切换到下一进度状态，或播放能落到下一状态的短 transition。
- 变化应该自然、短促、可感知：叹气、抬眼、肩膀松动、手部变化、身体回正、机器灯光反照、环境回暖。

### 6.4 音效与触感

- 点击命中：短促纸张、机器、电流、戳章音。
- 找错或无效点击：轻微空响，不惩罚玩家。
- 完成：机器稳定运转 + 纸袋封口音。
- 移动端可接入平台震动，强度要轻。

## 7. 关卡内容范围

MVP 第一阶段只做第一关：

- 场景：字节跳桶开放办公区。
- 主题：工作压力、踏空焦虑、老板压迫、轻度暴富幻想。
- 线索：亏损曲线、半截金价走势、owner 意识消息、花呗便利贴、刮刮泪。
- 不加入朋友圈收益墙、财神庙、彩票店完整柜台、AI 发布会主舞台等后续关卡主体。

完整 7 关剧情以 `docs/level-campaign.md` 为准。

## 8. 平台与后端需求

### 8.1 本地原型

- 本地使用 Vite dev server。
- 本地存档使用 localStorage。
- 本地排行榜可继续模拟。

### 8.2 微信/抖音小游戏方向

- 抽象平台桥接层，隔离 Web、微信、抖音 API。
- 关卡进度、广告观看、分享、排行榜、音频解锁应走统一接口。
- 广告逻辑不要嵌在 Phaser scene 内，应通过 platform bridge 调用。

### 8.3 排行榜与广告

- 排行榜是关卡外系统，不能出现在第一关完成页主流程。
- 广告位仅用于提示、双倍奖励或复活类场景，不能破坏第一关找线索节奏。
- 广告下发逻辑要记录每日次数、冷却和奖励倍率。

## 9. 提交与发布流程

使用 `scripts/commit-push.sh` 或 `npm run ship -- "message"` 作为标准提交脚本。

脚本行为：

- 自动进入仓库根目录。
- 确保远端使用 SSH 地址。
- fetch 当前分支。
- 如果远端分支存在，执行 `git pull --rebase --autostash`。
- 默认运行 `npm test`、`npm run build`、`npm run test:e2e`。
- `git add -A`，提交代码、文档和 public assets。
- push 当前分支；失败时 rebase 后重试。

快速跳过验证可使用：

```bash
SKIP_VERIFY=1 npm run ship -- "message"
```

正式合并前不建议跳过验证。

## 10. 验收标准

### 10.1 功能验收

- 第一关可开始、点击 5 个线索、显示收集进度、进入完成态、进入下一关占位。
- 默认不暴露热区。
- 点击反馈准确出现在被点击物件附近。
- 每找到一个新线索，主角或代理角色切换到下一进度状态。
- 完成态不展示处理线、资源卡、好友榜或设施升级。

### 10.2 视觉验收

- 画面统一，不出现 PNG 背景拼贴扁平 SVG 的割裂感。
- 人物状态变化来自素材，不出现补丁式五官。
- 每个关卡的主角或代理角色在 `progress-0` 到 `progress-N` 每个进度状态中必须有可见的表情或身体动作差异；只有环境变暖、机器亮起或文字变化不算通过。
- 移动端文字不裁切，按钮不挤压，场景主体可见。
- UI 看起来像游戏，不像后台。

### 10.3 技术验收

- Phaser scene 可独立运行在 Vite 本地开发环境。
- `npm test` 通过。
- `npm run build` 通过。
- `npm run test:e2e` 通过。
- Playwright 输出桌面与移动端关键截图。

## 11. 分阶段实施

### 阶段 1：Phaser 骨架

- 安装并接入 Phaser 3。
- 新增 boot/preload/office scene。
- 复刻第一关背景、热区、进度和命中逻辑。

### 阶段 2：第一关资产重构

- 拆分背景、线索、机器、人物状态。
- 完成 5 个线索命中特效。
- 完成人物 `N + 1` 个进度状态反馈，禁止 CSS 画脸或背景块动画。

### 阶段 3：关卡完成态

- 完成证据袋、机器稳定、下一关入口。
- 移除/隔离所有后台系统入口。
- 完成桌面/移动端视觉审核。

### 阶段 4：平台化准备

- 平台桥接层接入广告、分享、存档、排行榜抽象。
- 为微信/抖音小游戏构建目标做适配评估。

## 12. 明确不做

- 不继续用 CSS 在 raster 人脸上画表情。
- 不把第一关做成朋友圈、财神庙、AI 发布会素材大杂烩。
- 不在关卡完成页塞处理线、资源卡、排行榜、设施升级。
- 不用 hover 明示答案。
- 不把正式素材做成风格不一致的 SVG/PNG 拼贴。
